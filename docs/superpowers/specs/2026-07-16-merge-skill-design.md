# /merge skill — design

Date: 2026-07-16
Status: pending spec review
Decision trail: Alex, 2026-07-16 — "when working with features/worktrees
we should have a /merge command that will take care of everything that
needs for the merge to succeed". Fourth and last feature of the post-batch
arc. Shape approved via AskUserQuestion: a cs-shipped SKILL (agentic — it
can diagnose a failing gate instead of stopping), over a deterministic cs
verb or a verb+skill split. Scope approved: GENERIC WITH DISCOVERY — the
skill discovers each repo's gates from the repo itself; cs's own rules
(build.sh before tests, deploy surfaces) keep living in cs's CLAUDE.md,
not in the skill. (Final-review correction: this repo's own gate
documentation lives in CONTRIBUTING.md, not CLAUDE.md — discovery reads
the project's instruction files broadly.)

## Context and goal

The merge ritual this arc ran eight times — gates on the branch, merge
--no-ff, gates again on the merged result, deploy, cleanup — exists only
in session ledgers and the operator's head. Task worktrees have a verb
(`cs <base> --merge <task>`) that fuses records and merges, but nothing
runs gates around it. The goal: `/merge` encodes the whole closing ritual
as a skill shipped by cs to `~/.claude/skills/`, available in every
session, correct in any repo.

## Decisions

1. **A skill, not a verb.** The failure mode of merging is a failing
   gate; a shell verb can only stop, a skill instructs Claude to
   root-cause (the repo's systematic-debugging rules apply). The skill
   NEVER bypasses a gate, never forces, never deletes a branch after a
   failure.
2. **Generic with discovery.** Gate discovery order, taught by the
   skill: (a) the project's instruction files govern absolutely —
   CLAUDE.md and anything it imports, CONTRIBUTING.md, and the README's
   development section — for build steps, test commands, generated
   artifacts, deploy steps; (b) failing that, conventional entry points in
   preference order: `tests/run_all.sh`, `Makefile` test target,
   `package.json` scripts.test, `cargo test`, `go test ./...`,
   `pytest`; (c) nothing found → ask the user once for the gate command
   and remember it for the rest of the conversation. A repo whose
   instructions document a build step for generated artifacts (cs's
   `./build.sh` regenerating `bin/cs`) gets the build run BEFORE tests
   and the generated file committed with the branch, per those
   instructions.
3. **Two merge contexts, detected up front:**
   - **Task worktree session** (the session directory is
     `<base>@<task>`, or `git rev-parse --git-dir` differs from
     `--git-common-dir` inside a cs worktree session): gates run in the
     worktree; the merge itself is `cs <base> --merge <task>` (record
     fusion + git merge + worktree cleanup, already implemented); the
     post-merge gates run in the base session checkout. (Final-review
     correction: the verb refuses while either session's lock is live,
     so from inside the worktree session the skill runs the preflight
     gates and then hands the verb to the user — close the session, run
     it from a free terminal — rather than running it itself.)
   - **Feature branch** in an ordinary checkout: preflight gates on the
     branch → `git merge --no-ff <branch>` into the base branch (the
     branch it forked from; `git merge-base` + the repo's default branch
     resolve it, ask when ambiguous) → gates again on the merged result
     → delete the merged branch.
   - Neither (on the default branch with nothing to merge) → say so and
     stop.
4. **Safety rails, non-negotiable, stated in the skill:** never merge
   with uncommitted changes (offer to commit first and stop if
   declined); never push — publishing is a separate user decision the
   skill must not make; never delete a branch until the post-merge gates
   are green; a failing post-merge gate leaves the merge commit in place
   and reports — reverting main is the user's call, not the skill's.
5. **After the gates: offers, not actions.** Deploy only if the repo's
   instructions document a deploy step, offered via one question, never
   run unprompted. In a cs session, offer `/checkpoint
   <feature>-merged` after a green merge. Both are offers because they
   are outward-facing or state-changing beyond the merge itself.
6. **Registration and deploy:** `skills/merge/SKILL.md`; `merge` added
   to `CS_SKILLS` in BOTH manifests (`lib/00-header.sh` and
   `install.sh`, which carry KEEP-IN-SYNC comments); deployed to
   `~/.claude/skills/merge/` by install/update. `bin/cs` regenerated by
   `./build.sh` (the manifest lives in a fragment) and committed with
   the fragments.

## SKILL.md outline (the plan carries the full text)

Frontmatter: `name: merge`, description triggering on "merge this
branch / merge the worktree / close out this feature" requests. Body
sections: Prerequisites (clean tree, resolve the context per Decision
3); Discover the gates (Decision 2 order, verbatim); The ritual
(preflight gates → merge → post-merge gates → cleanup, with the exact
git commands and the worktree wrap); When a gate fails (diagnose,
root-cause, fix-forward on the branch, never bypass; re-run from the
top after any fix); Offers (Decision 5); Never (the Decision 4 rails as
a bullet list).

## Out of scope

- No review pipeline: /merge assumes the branch is already reviewed to
  the user's standard; it is the mechanical closer, not a quality gate.
- No push, no release, no changelog authoring.
- No changes to `cs <base> --merge` itself.

## Testing

New suite `tests/test_merge_skill.sh` (auto-discovered by
`tests/run_all.sh`'s `test_*.sh` glob), mirroring the rotate skill's
registration cycle. Every assert `|| return 1`; `report_results` last.

1. `skills/merge/SKILL.md` exists, opens with `---` frontmatter,
   `name: merge`, has a `description:`.
2. Registered in both manifests: `grep -A 6 '^CS_SKILLS=('` on
   `lib/00-header.sh` and `install.sh` finds `merge`.
3. Contract text pinned (the agentic behavior is not unit-testable; the
   taught contract is): the skill contains `--no-ff`, runs gates before
   AND after (assert on the phrases `preflight` and `merged result`),
   guards the clean tree (`uncommitted`), wraps the worktree verb
   (`--merge`), and states the rails `never push` and the
   branch-deletion condition (`until the post-merge gates are green`).
4. `bin/cs` manifest sync: the built `bin/cs` contains `merge` inside
   its `CS_SKILLS=(` block (catches a fragment edit without a rebuild).

## Files

- `skills/merge/SKILL.md` — the skill (full text in the plan).
- `lib/00-header.sh`, `install.sh` — CS_SKILLS entries.
- `bin/cs` — regenerated by `./build.sh`.
- `tests/test_merge_skill.sh` — the four cases.
- `README.md` — add `merge` to the installed-skills line ("Adds
  `/summary`, `/checkpoint`, ... and the `store-secret` and
  `prose-hygiene` skills"), and one sentence in the worktrees section
  pointing at `/merge` as the gated closer.

Deploy surfaces: `~/.local/bin/cs` (manifest fragment changed) and
`~/.claude/skills/merge/`.
