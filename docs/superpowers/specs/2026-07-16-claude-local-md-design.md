# CLAUDE.local.md as the session protocol target — design

Date: 2026-07-16
Status: pending spec review
Decision trail: Alex, 2026-07-16 — "can our tool create or update
CLAUDE.local.md instead of CLAUDE.md? it seems the safer choice". Fifth
feature of the post-batch arc. Verified against the installed CC 2.1.211
bundle: CLAUDE.local.md is fully supported — auto-loaded alongside
CLAUDE.md, with a documented worktree stub-import pattern. Decisions via
AskUserQuestion: full lazy migration for existing sessions; the emptied
CLAUDE.md is removed only when nothing but cs content was in it.

## Context and goal

cs writes its Session Documentation Protocol into each session's
`CLAUDE.md` (create path overwrites; lazy-migration Phase 5 appends to a
user-authored file when the sentinel is absent). Appending to a file the
USER owns is the riskiest thing cs does to user content, and a tracked
`CLAUDE.md` invites cross-machine merge noise for what is really
tool-generated scaffolding. The goal: cs writes its protocol to
`CLAUDE.local.md` — machine-local, gitignored, regenerated by cs itself —
and never touches `CLAUDE.md` again. Claude Code loads both files, so
the protocol keeps working and nothing is lost even mid-transition.

## Decisions

1. **Target file: `CLAUDE.local.md` at the session root.** The create
   path (`write_session_claude_md`, called from migration when the
   protocol is absent everywhere) writes the SAME template there. The
   three managed sentinel sections (`cs:session-protocol`,
   `cs:memory-note`, `cs:wrap-cues`) and all their semantics — including
   the wrap-cues tombstone convention (sentinel presence = managed, do
   not re-add) — move wholesale; their text does not change.
2. **Gitignored, per machine.** The session `.gitignore` gains a
   `CLAUDE.local.md` entry (create path and migration both ensure it,
   idempotently). Each machine's cs regenerates the file at launch via
   the same lazy migration that maintains it today; nothing about it
   syncs. This satisfies the multi-user classification rule:
   machine-local state stays out of the session git.
3. **Full lazy migration of existing sessions**, as a new migrate phase
   running before today's Phase 5 logic:
   - `CLAUDE.md` starts with `<!-- cs:session-protocol -->` (the pure
     cs-written file — the overwhelmingly common case): move the entire
     file to `CLAUDE.local.md` and delete `CLAUDE.md` (plain `rm`; the
     deletion rides the working tree like every other lazy-migration
     edit, protected by autosave — cs has no internal commit machinery
     and this phase does not add one).
   - `CLAUDE.md` contains a cs sentinel but does not start with one
     (user head + appended cs sections): move everything from the FIRST
     `<!-- cs:` sentinel line to EOF into `CLAUDE.local.md`, keep the
     user's head in `CLAUDE.md`, delete nothing. Caveat, accepted: any
     user content that was interleaved AFTER cs's first sentinel rides
     along into the local file — visible, still auto-loaded by Claude
     Code, nothing lost functionally.
   - `CLAUDE.md` references `.cs/` but carries no sentinel (a
     pre-sentinel-era template): leave the session entirely on
     `CLAUDE.md` — extraction cannot be surgical without sentinels, and
     writing a second protocol file would duplicate instructions. Rare,
     documented limitation.
   - No `CLAUDE.md` and no `CLAUDE.local.md`: create path writes
     `CLAUDE.local.md`.
   - `CLAUDE.local.md` already present with the sentinel: no-op
     (idempotence; second run of any case above is silent).
   - Existing Phase 5 (protocol append), Phase 9 (memory-note
     management), and Phase 10 (wrap-cues) retarget to the protocol
     file — `CLAUDE.local.md` when present, else the legacy `CLAUDE.md`
     for pre-sentinel-era sessions — and never write to a user's
     `CLAUDE.md` again.
4. **Determinism across machines.** The extraction is a pure function
   of the file's content (sentinel line boundaries, no timestamps, no
   local state), so two clones migrating the same tracked `CLAUDE.md`
   produce byte-identical edits that merge cleanly — the same rule the
   discoveries fold follows.
5. **Worktree sessions get their own copy.** `CLAUDE.local.md` is
   gitignored, so a new worktree does NOT inherit it through git: the
   worktree bootstrap must invoke the create path in the new worktree
   (today it "leaves the checkout's CLAUDE.md alone"; it now writes the
   worktree's `CLAUDE.local.md`). It writes the FILE only — never a
   `.gitignore` entry: ignored-mode worktrees check out project repos,
   and mutating a project's tracked `.gitignore` would pollute the task
   branch.
6. **Known consequence for this repo (the cs dev session):** its
   tracked `CLAUDE.md` is a pure cs template and will migrate — moved
   to the gitignored local file, the tracked file deleted from the
   project repo on this branch. That is the intended end state
   (session scaffolding out of the public repo), stated here so the
   final review treats the deletion as designed, not collateral.

## Out of scope

- No changes to the template text or the managed-section semantics.
- No `~/.claude/<project>` import-stub pattern (the bundle documents
  one for sharing personal content across worktrees; cs regenerates
  instead — simpler, no home-directory coupling).
- No doctor checks (none reference CLAUDE.md today).
- No handling of project CLAUDE.md files cs never wrote.

## Testing

Extend `tests/test_migrate_claude_md.sh` (the suite that owns this
surface). bash 3.2 + BSD; every assert `|| return 1`; `report_results`
last. Cases:

1. Create path: a new session gets `CLAUDE.local.md` with the
   `cs:session-protocol` sentinel, NO `CLAUDE.md`, and a `.gitignore`
   containing `CLAUDE.local.md`.
2. Pure-cs migration: a session whose `CLAUDE.md` starts with the
   sentinel → content now in `CLAUDE.local.md`, `CLAUDE.md` gone.
3. Mixed-file migration: user head + appended cs sections → head stays
   in `CLAUDE.md`, sentinel-to-EOF lands in `CLAUDE.local.md`, both
   sentinels (`session-protocol`, `memory-note`) present in the local
   file, user text absent from it.
4. Pre-sentinel template (`.cs/` reference, no sentinel): untouched, no
   `CLAUDE.local.md` created.
5. Idempotence: running migration twice over case 2 and case 3 changes
   nothing on the second pass (byte-compare both files).
6. Phase 9 retarget: a migrated session missing the memory-note gets it
   added to `CLAUDE.local.md`, and `CLAUDE.md` is not recreated.
7. Worktree bootstrap: creating a task worktree writes the worktree's
   own `CLAUDE.local.md` and `.gitignore` entry.
8. Gitignore idempotence: migration over a session that already has a
   `.gitignore` with the entry does not duplicate the line.

## Files

- `lib/35-claudemd.sh` — create path retarget; the extraction helper.
- `lib/45-migrate.sh` — the new migration phase; Phase 5/9 retargets.
- `lib/30-worktree.sh` — worktree bootstrap writes the local file.
- `lib/85-adopt-uninstall.sh` — `create_session_gitignore()` (the

  `ensure_cs_gitignore_entries()` (idempotent backfill on existing
  sessions, already called from migration).
- `hooks/session-start.sh` — its context text says "See CLAUDE.md in
  the session directory" (twice, lines ~138/140); both references
  become "CLAUDE.local.md". Hooks deploy surface therefore included.
- `bin/cs` — regenerated by `./build.sh`.
- `tests/test_migrate_claude_md.sh` — the eight cases.
- `README.md`, `docs/session-layout.md` — CLAUDE.local.md replaces
  CLAUDE.md in the session-layout story.

Deploy surfaces: `~/.local/bin/cs` and
